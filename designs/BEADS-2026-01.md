## Appendix: MOOLLM Beads vs Gastown Beads

> *"Compatible extension, not replacement. Same principle as Anthropic skills."*

### Schema Comparison

| Aspect | Gastown | MOOLLM | Compatibility |
|--------|---------|--------|---------------|
| **Storage** | JSONL (git-backed) | YAML (Markdown-embedded) | ‚úÖ Different but equivalent |
| **ID Format** | `prefix-5char` (`gt-abc12`) | `prefix-NNNNN` (`moollm-001`) | ‚úÖ Both use prefix routing |
| **Status** | `open`, `in_progress`, `closed` | `pending`, `in-progress`, `done`, `blocked` | ‚ö†Ô∏è Mapping needed |
| **Priority** | Implicit (via labels) | Explicit: `critical`, `high`, `medium`, `low` | üÜï MOOLLM extension |
| **Dependencies** | `bd dep add <child> <parent>` | `depends_on: []` | ‚úÖ Same concept, inline |
| **Labels/Tags** | Labels (system) | `tags: []` | ‚úÖ Equivalent |
| **Source** | Implicit (via rig/prefix) | `source: "origin doc"` | üÜï MOOLLM extension |

### What Gastown Has That MOOLLM Doesn't (Yet)

| Feature | Gastown | MOOLLM Equivalent | Status |
|---------|---------|-------------------|--------|
| **Formulas** | TOML workflow templates | Skills with templates | Could map |
| **Molecules** | Durable chained workflows | Task sequences in LOG.md | Different approach |
| **Wisps** | Ephemeral beads | Not needed (LLM context is ephemeral) | N/A |
| **Hooks** | Git worktree persistence | Three-tier persistence | Different mechanism |
| **Routing** | Prefix ‚Üí database | K-lines + directories | Semantic vs structural |
| **CLI** | `bd` command | None (LLM interprets YAML directly) | Design choice |

### What MOOLLM Adds (Beyond "Not Being Insane")

| Feature | Why It's Distinct | Gastown Parallel |
|---------|-------------------|------------------|
| **Semantic fields** | `description`, `summary`, `deliverables` as rich YAML | Labels/description |
| **Skill integration** | Beads can reference `skills/*` deliverables | Formulas reference scripts |
| **Priority explicit** | First-class priority, not label hacks | Could add via labels |
| **Blocked status** | Explicit blockers, not just dependencies | Could add via labels |
| **Source tracking** | "Where did this task come from?" | Implicit via rig |
| **Markdown-first** | Human-readable document, not database | JSONL is for tools |

### The Key Insight: Different Audiences

```yaml
gastown_beads:
  audience: "Go CLI tools (bd, gt)"
  format: "JSONL ‚Äî machine-first, git-versioned"
  persistence: "Append-only ledger"
  routing: "Prefix ‚Üí database file"
  
moollm_beads:
  audience: "LLM reading a design doc"
  format: "YAML in Markdown ‚Äî human-first, LLM-parseable"
  persistence: "Document (read the whole thing)"
  routing: "K-lines + semantic associations"
```

### Compatible Extension Strategy (Like Anthropic Skills)

**Principle**: Any valid Gastown bead concept works in MOOLLM. MOOLLM beads can be "downgraded" to Gastown format.

**Mapping MOOLLM ‚Üí Gastown**:

```yaml
# MOOLLM bead
bead:
  id: moollm-006
  title: "Create no-ai-sycophancy skill"
  status: pending
  priority: high
  depends_on: [moollm-001]
  tags: [quality, behavior]

# Gastown equivalent (bd create)
# bd create --title="Create no-ai-sycophancy skill" \
#           --type=task \
#           --labels="priority:high,quality,behavior"
# bd dep add moollm-006 moollm-001
```

**Mapping Gastown ‚Üí MOOLLM**:

```yaml
# Gastown issue (from bd show)
# gt-abc12: "Fix auth bug" (open, labels: security, urgent)

# MOOLLM representation
bead:
  id: gt-abc12
  title: "Fix auth bug"
  status: pending
  priority: high  # inferred from "urgent" label
  tags: [security]
```

### Why MOOLLM Beads Are YAML-in-Markdown

| Reason | Explanation |
|--------|-------------|
| **LLM-native** | LLMs read YAML better than JSONL |
| **Human-friendly** | Design docs are readable documents |
| **Context budget** | Single file loads entire backlog |
| **Semantic richness** | Descriptions, summaries, lineage embedded |
| **No tooling required** | LLM parses directly, no `bd` CLI |

### The "Just Not Being Insane" Parts

These are standard good practices, not MOOLLM inventions:

- IDs with prefixes (namespacing)
- Status tracking (any issue tracker)
- Dependencies (standard graph)
- Tags/labels (categorization)
- Git-backed (version control)

### The Genuinely Novel Parts

| Innovation | Description |
|------------|-------------|
| **Play ‚Üí Learn ‚Üí Lift** | Beads become schemas become skills |
| **Source field** | Explicit provenance ("where did this come from?") |
| **Deliverables list** | What will this bead produce? |
| **Skill cross-references** | Beads reference `skills/*` paths |
| **Markdown embedding** | Beads are documentation, not database rows |
| **K-line activation** | Bead IDs are semantic activators |
